# 実装計画: 9:16 正規ページ・編集/閲覧互換・RTL/LTR・言語別ページの維持

## 0. 目的と非目的

**目的**

- **9:16 の正規フレーム**を、エディター・Press（WebP 焼き込み）・ビューワーで**同じ定義**として扱い、**はみ出し・縦横比崩れ**を起こしにくくする。
- **言語ごとの WebP**（`content.backgrounds[lang]`）と **`languageConfigs[lang].pageDirection`（RTL/LTR）** を**変更せず**に、表示幾何だけを堅くする。

**非目的（本計画の外に出す）**

- EPUB 的リフロー、ビューワー側での rich テキスト組版の復活。
- サムネイル用 16:9 パイプラインの全面置換（`firebase.js` の切り抜きサムネはページとは別比率。必要なら別タスク）。

## 1. 現状整理（コードベース）

| 領域 | 現状 | ギャップ / リスク |
|------|------|-------------------|
| エディター | `EDITOR_FRAME_WIDTH/HEIGHT = 360/640`（`app.js`） | マジックナンバーが分散。他ファイルとズレると再発。 |
| ビューワー | `resizeCanvas` で `360/640` と `stage` の `translate + scale`（`viewer.js`） | 画像が `cover` 相当で描画される箇所と `contain` の不一致がないか要確認。ズーム/パンと競合時のクリップ。 |
| Press | `_renderPageToWebP` が `baseW=360` から `targetW/H` へスケール（`press.js`） | **既定の `targetW/H` が 1080×1920 未満になっていないか**を確認し、最低ラインをコードと UI で強制。多解像度プリセットの一元化。 |
| データ | Gen3: `backgrounds` は言語キー、ナビは `pageDirection`（`viewer.js`） | データモデル変更不要。 |

## 2. 設計原則（守ると壊れないもの）

1. **正規論理解像度**: レイアウト・吹き出し座標・背景オフセットの**唯一の座標系**は `W0×H0`（現状 `360×640`）。**非等方スケール禁止**（表示も書き出しも `uniform scale`）。
2. **配信ビットマップ（物理ピクセル）の最低ライン**: **1080×1920（9:16）を最低**とする。論理 `360×640` との対応は **`k = 3` の等倍スケール**（`1080 = 360×3`, `1920 = 640×3`）を既定とし、非整数倍によるにじみ・ずれを避ける。より高品質が必要な場合のみ **`k = 4`（1440×2560）** 等をオプションで提供する。
3. **言語間の倍率 `k`**: 各言語の WebP は原則 **同じ `k`**。品質プリセットは「解像度（k）」で揃え、縦横比は常に 9:16。
4. **RTL/LTR**: **ナビと UI の方向**に使い、**ビットマップを水平反転しない**（縦書き・見開きは画像に焼かれた結果をそのまま表示）。
5. **はみ出し対策**: ビューワーは **枠に `contain`**。**溢れは組版・Press 前**で検知（既存の overflow / 次ページ分割と連携）。

## 3. フェーズ別タスク

### フェーズ A — 定数の単一ソース化（低リスク・最優先）

| # | タスク | 成果物 |
|---|--------|--------|
| A1 | `CANONICAL_PAGE_WIDTH`, `CANONICAL_PAGE_HEIGHT`, `CANONICAL_PAGE_ASPECT`（= W/H）を **1 モジュール**に定義（例: `js/page-geometry.js` または `js/constants-layout.js`）。 | 新規小ファイル + import |
| A2 | `app.js` の `EDITOR_FRAME_*`、`viewer.js` の `360/640`、`press.js` の `baseW/baseH` を **すべてその定数から参照**。 | 重複排除 |
| A3 | `CLAUDE.md` / `file-format-spec.md` の「9:16」の説明を **「正規論理 360×640（変更時は定数参照）」** に一言追記。 | ドキュメント同期 |

**受け入れ条件**: リポジトリ内に `360` と `640` のリテラルが散在しない（定数以外はテスト用のみ等）。

### フェーズ B — ビューワー表示の「枠内 guarantee」

| # | タスク | 成果物 |
|---|--------|--------|
| B1 | `content-stage` 内のメイン画像の描画方針を **`object-fit: contain` + 明示的 360×640（論理）枠**に統一。`cover` によるクロップが残っていれば **意図をドキュメント化**したうえで `contain` に寄せるか判断。 | `viewer.css` / `viewer.html` / `viewer.js` の最小修正 |
| B2 | `resizeCanvas` の計算を **「外枠は画面に収める 9:16 + 内側は論理 360×640 に uniform scale」**としてコメントと式を読みやすく整理（挙動は変えないことを原則）。 | 可読性・将来の `dvh` 対応余地 |

## 4. リスクと緩和

| リスク | 緩和 |
|--------|------|
| `object-fit: cover` に依存していた見た目が変わる | フェーズ B で「意図的クロップが必要な作品」があるか確認。**必要ならエディター側でトリミングを完結**させ、配信は contain 前提に寄せる。 |
| 高解像度のみの端末でぼやける | 既定を **k=3（1080×1920）** にしたうえで、オプションで **k=4** 等を Press に用意（品質プリセットと統合）。 |
| サムネイル比率（16:9）とページ（9:16）の混同 | コードコメントと変数名で **THUMB vs PAGE** を分離。 |

## 5. 推奨スケジュール（目安）

1. **スプリント 1**: フェーズ A（定数化のみ）— レビュー容易、挙動変更最小。  
2. **スプリント 2**: フェーズ B + E のビューワー回帰。  
3. **スプリント 3**: フェーズ C + D + 残り E。

## 6. 完了の定義（Definition of Done）

- 正規ページ寸法が **単一モジュール**から参照されている。
- ビューワーで **9:16 外枠 + contain** が仕様として文書化され、主要端末で手動確認済み。
- RTL/LTR・言語別 `backgrounds` の既存データが **マイグレーション不要**で動作。
- Press 出力の解像度が **論理フレームの整数倍**で追跡可能。
- **配信用 WebP の物理解像度が 1080×1920 未満にならない**（UI・既定値・エクスポート経路で保証）。

## 7. 本計画の前後で優先したい別課題（リポジトリ現状ベース）

9:16 定数化・1080 最低ラインは **表示品質と編集/配信の一致**に直結するが、**本番公開・運用・スケール**では次が先に詰まりやすい。並行は可能だが、**リリース判断では先に着手したい順**の目安。

| 優先度 | 課題 | 理由 |
|--------|------|------|
| **高** | **Firestore / Storage（および Pages Function 前提）のセキュリティルール確定** | 公開ビューワ・非公開・ゲスト有無が絡むと、仕様より先に**データ漏えい・誤公開**がブロッカーになり得る。 |
| **高** | **Cloudflare 本番運用の一本化**（Pages / R2 バインディング、`VITE_STORAGE_BACKEND`、プレビュー環境） | 画像 1080×1920 化は **転送量・コスト・キャッシュ**が増える。インフラが曖昧だと検証とコスト試算が回らない。 |
| **高** | **`firebase.js`（等）の設定 `.env` 化**（`AGENTS.md` 記載の未対応） | 環境ごとの切替・レビュー・CI と直結。本番キー運用の前提。 |
| **中** | **公開 / 非公開の権限モデル明文化と実装ギャップ解消**（`PROJECT_STATUS` 既知） | クリエイター体験とサポート負荷に直結。 |
| **中** | **プロジェクト一覧のページネーション**（`projects.js` 全件取得） | ユーザー数が増えると先に効いてくる。9:16 実装とは独立に進められる。 |
| **中〜長** | **コンテンツ監査・停止・削除の最小運用** | 公開サービス化のタイミングで必要。レイアウト実装と並列可。 |
| **低（同一スプリント内でよい）** | 組版プリセット（禁則等）の精緻化 | 1080 化と相性はよいが、**幾何の一本化が先**でもよい。 |

**結論**: **9:16 / 1080×1920 の実装を止めるほどの技術的依存は少ない**。ただし **セキュリティルール・本番インフラ・設定外出し**は、プロダクトを外に出すなら **同じスプリント帯で優先度を上げた方がよい**。文庫・小説向けなら、**Press 既定を 1080 に上げる変更**は転送量が跳ねるため、**R2 / 帯域の見積もり**とセットで進めると安全。
