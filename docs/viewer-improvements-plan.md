# DSF Viewer: 現状の課題と改善案 (Viewer Agent Analysis)

Viewer Agent として `viewer.html`, `viewer.js`, `viewer.css` のコードベースを精査し、現在の問題点と今後の改善案を整理しました。
アーキテクチャ全体としては SPA（Single Page Application）として優秀な構造を持っていますが、過去の負債やモバイル特有のUI制御においていくつかの課題が見受けられます。

---

## 1. 現状のコードベースの課題

### 課題 A: レガシーデータ対応による条件分岐の肥大化 (`viewer.js`)
*   **症状**: `refresh()` やレイアウト計算処理の中に、v4以前のデータ構造（`state.sections`）とv5以降のデータ構造（`state.pages`）の両方に対応するための `if/else` 分岐が多数混在しています。
*   **影響**: コードの可読性が著しく低下しており、新機能（ARなど）を追加する際のバグの温床になりやすい状態です。
*   **改善案 (優先度: 高)**: 
    *   読み込み時 (`loadProjectData`) に、**すべての古いデータをv5フォーマット（`pages`ベース）に強制マイグレーション**する処理を完全に移譲し、レンダリングロジックからはレガシー対応の分岐をすべて削除すべきです。

### 課題 B: タッチイベントとジェスチャー制御の複雑さ
*   **症状**: モバイル特有の挙動（ダブルタップによるUIトグル、ロングプレス）を制御するために、`handleTouchStart_Fixed`, `handleTouchMove_Fixed` などのラッパー関数が多用されていますが、ネイティブのスクロール制御と競合しがちです。
*   **影響**: OSアップデートやブラウザの仕様変更に弱く、意図しないタイミングでUIが表示されたり、スワイプが誤爆するリスクがあります。
*   **改善案 (優先度: 中)**:
    *   カスタムのタッチ制御を減らし、CSSの `touch-action` や `Pointer Event API` にモダン化することで、コードを劇的に軽量化・安定化させることができます。

### 課題 C: テキスト余白計算のハードコーディング
*   **症状**: `getVerticalTextPadding()` などの関数内で、日本語の縦書きフォントの余白を計算するためにハードコードされたマジックナンバー（例: 最大40pxのクランプ等）が含まれています。
*   **影響**: 特定のデバイス解像度や新しいフォントプリセットを追加した場合に、文字のセンタリングが微妙にズレる可能性があります。
*   **改善案 (優先度: 低)**:
    *   JSでの絶対値計算に依存せず、Flexbox や CSS Grid などの最新のレイアウトモジュールに配置計算を委譲する設計（純粋なWeb表現）へリファクタリングする。

---

## 2. パフォーマンス・UXの改善案（ユーザー視点）

### 改善 1: プリロード（先読み）対象の拡大
現在、現在のページの前後1ページ分（`preloadSurroundingImages`）の画像を先読みして最適化するロジックが入っていますが、対象が「画像」のみに限定されています。
*   **提案**: フォントファイル（Noto Sans/Serif系）など、表示に時間のかかるアセットも最初からプリロード対象に含め、「文字の表示遅れ（FOUT）」を完全になくす。

### 改善 2: ページ遷移アニメーションの最適化
現状、前後のページへの遷移は瞬時に内容が切り替わる仕様（カットイン）になっています。
*   **提案**: （漫画形式の場合は現状のカットインが最適ですが）テキスト主体の章や節の遷移の際には、電子書籍のように非常に短いフェード遷移（0.15秒程度）を挟むことで、より高級なリーダーアプリとしての質感が向上します。

### 改善 3: アクセシビリティ (a11y) の強化
*   **提案**: `aria-label` などの一部対応は入っていますが、キーボード操作用のフォーカスリング設計や、読み上げソフト（VoiceOver 等）が確実にテキストだけを拾えるような `role="article"` 属性の付与など、Web標準のアクセシビリティ対応を強化することで、公共性の高いフォーマットとしての価値が高まります。
