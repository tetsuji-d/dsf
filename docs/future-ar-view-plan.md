# DSF Viewer: AR View 統合構想 (Future Plan)

このドキュメントは、DSF (Digital Smart Format) の Viewer アプリケーション（`viewer.html`）に将来的に実装予定の「AR VIEW」機能に関する構想、要求仕様、および技術的アプローチをまとめたものです。

---

## 1. コンセプト概要

ユーザーが現在閲覧しているDSFのページ（マンガ、地図、インフォグラフィックなど）を、現実空間の背景と融合させ、直感的かつ空間的な閲覧体験を提供するモードです。通常の2D閲覧モードからシームレスに切り替え可能であることを前提とします。

### 主な想定機能

*   **AR ZOOM**:
    空間上に仮想的に配置されたDSFページに対し、スマートフォンを近づけることで特定の部分をシームレスに拡大表示し、引くことで見開きや複数ページ全体を俯瞰できる機能。
*   **AR GUIDE MAP**:
    現在地の実際の風景（カメラ映像）の足元（空間地平）にマップページを重畳表示。スマートフォンのコンパス機能と連動してページが実際の方角を向き、現在地ピンと合わせて直感的なナビゲーションを提供する機能。

---

## 2. ユーザー体験 (UX) フロー

1.  **モード切り替え (In)**:
    *   通常の2D閲覧中、UIオーバーレイ上に表示される「AR」ボタンをタップする。
    *   （初回のみ）カメラおよびデバイスモーションセンサーのアクセス許可を要求。
    *   画面の黒背景がフェードアウトし、デバイスの背面カメラ映像がフェードイン。ロード画面等を挟まず、シームレスに透過AR空間へ移行。
2.  **空間でのインタラクション**:
    *   ジャイロセンサーやWebXR空間認識により、デバイスを動かすことで視点が変化する。
3.  **モード切り替え (Out)**:
    *   UI上の「✕」または「2Dに戻る」ボタンをタップ。
    *   カメラ映像がフェードアウトし、元の2Dスクロール閲覧状態へ瞬時に戻る。

---

## 3. 技術的アプローチと実現フェーズ

実現には段階的なアプローチが推奨されます。

### Phase 1: 疑似 AR モード (ジャイロ連動 2.5D)

本格的な3D空間認識を用いず、既存のDOM描画の枠組みで実現するアプローチ。
*   **技術要素**:
    *   `navigator.mediaDevices.getUserMedia()`: 背面カメラ映像の取得し、`<video>` エレメントとして Viewer の最背面にフルスクリーン配置。
    *   `DeviceOrientationEvent`: スマートフォンの傾き（ジャイロ）やコンパス方位を取得。
    *   CSS `transform: perspective() rotateX() rotateY() translateZ()`: ジャイロの値に連動してコンテンツ全体を傾けたり動かしたりして、空間に浮いているような錯覚を作る。
*   **メリット**: 実装コストが低く、iOS/Android 問わず動作可能。現在のHTML/CSSベースの描画エンジン (`viewer.js`) をそのまま活かせる。
*   **制限**: 実際にカメラを近づけて（Z軸の移動）拡大することはできず、疑似的な表現にとどまる。

### Phase 2: WebXR への本格移行 (空間ピン留めと AR ZOOM)

空間を認識し、真のAR体験を提供するアプローチ。
*   **技術要素**:
    *   **WebXR Device API**: ブラウザ標準のAR/VR API。現実空間の平面認識 (Hit Test) や完全な6DoF（位置と傾きの両方）トラッキングを提供する。
    *   **Three.js / A-Frame**: 3Dグラフィックスの描画エンジン。
*   **実装上の課題と対応**:
    *   HTML要素（通常のテキストやフキダシ）を直接WebXR空間に置くことはできない。
    *   対策として、既存のDOM（ページ内容）を `html2canvas` 等で見えない裏側で一枚の高解像度テクスチャ画像にレンダリング（Rasterize）し、それを3D空間内の板（PlaneGeometry）に貼り付ける手法をとる。
*   **メリット**: 「スマホを近づけて拡大 (AR ZOOM)」や「床にマップを固定 (AR GUIDE MAP)」が完璧に実現できる。
*   **制限**: 現状、iOS Safari が WebXR Device API を標準サポートしていない（※将来対応する可能性あり）。

## 4. データモデル (憲法) への要求事項

エディタ側で「このページはAR対応か（あるいはARマップか）」を設定できるよう、`docs/data-model.md` へのプロパティ追加を Architect に要求する必要があります。

**追加想定プロパティ (Page Object 内)**:
*   `arMode`: `string` ("none", "gyro-map", "xr-space") - ARモードの種類
*   `arScale`: `number` - 空間上における初期スケール値
*   `geolocation`: `{lat, lng}` - (マップの場合) アンカーとなる現実の緯度経度

---

## 5. Viewer Agent 開発予定タスク

1.  **[Architectへ要求]**: データモデルへのAR関連プロパティの追加提案。
2.  **[UI実装]**: 該当プロパティを持つページでのみ表示される「ARモードトグルボタン」の追加。
3.  **[背景透過処理]**: ARボタン押下時に、Viewerキャンバスの黒背景を透明にし、`<video>`タグで背面カメラ映像を描画する仕組みの構築。
4.  **[センサ連携]**: Phase 1 として、`DeviceOrientationEvent` を監視し、キャンバス全体に `perspective` を効かせた CSS 3D Transforms を適用するスクリプトの実装。
